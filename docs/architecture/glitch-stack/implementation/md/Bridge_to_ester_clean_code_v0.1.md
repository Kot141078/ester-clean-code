## 1. Purpose

This document does **not** claim that the current repository already implements the full triad stack exactly as specified in the new concept documents.

Its purpose is narrower:

- identify where the current repository already exposes strong conceptual anchors,
- define where the new stack can attach without architectural violence,
- separate **existing implementation terrain** from **future insertion points**.

This is a bridge document, not a retrofit fantasy.

## 4. Bridge philosophy

The new conceptual stack should **not** be inserted as a decorative visualization layer.

It should be attached according to this rule:

- **Core remains runtime truth**
- **Research remains quarantined**
- **Walkthrough remains non-executive**
- **Witness remains evidence, not authority**
- **Challenge remains procedural, not theatrical**
- **Status algebra remains machine-checkable**

In practice, this means the bridge must preserve the repo's existing safety posture:
**fail-closed, explicit privilege, explicit budgets, witness-first execution.**

## 5. First-pass conceptual -> repository mapping

## 5.1 L4 Glitch Map

### Concept role
The place where execution collides with reality and yields a typed `GlitchNode` instead of speculative continuation.

### Likely repository attachment points
- `ESTER/` for runtime surfaces where action attempts originate
- `modules/` for subsystem-level failure emission
- `rules/` for lock-type interpretation
- `validator/` for transition refusal logic
- `middleware/` for enforcement at boundaries
- `observability/` and `metrics/` for event emission
- `tests/` for lock-trigger assertions

### Likely implementation shape
- typed lock enums
- runtime failure object
- fail-closed stop hooks
- event emission into witness / metrics path

### Important rule
`GlitchNode` should be generated by runtime collision, not by UI speculation.

## 5.2 Research Node / Backward Node

### Concept role
Quarantined memory object representing what the system could not do, what was missing, and what future evidence would be required to reopen the path.

### Likely repository attachment points
- `storage/` for persistent retention
- `schemas/` for formal object definition
- `thinking/` for reinterpretation and contextual revisit
- `volition/` for non-executive relation to intent

## 6. Proposed minimum implementation order

To preserve fail-closed discipline, implementation should proceed in this order:

### Phase 1 — state discipline first
1. status enums / algebra
2. legal transition guards
3. `GlitchNode` object model
4. test coverage for forbidden transitions

### Phase 2 — quarantine before cinema
5. `ResearchNode` / `BackwardNode` storage model
6. quarantine retrieval rules
7. runtime exclusion from execution paths

### Phase 3 — evidence before interface beauty
8. witness metadata schema
9. challenge / review objects
10. expiry and settlement mechanics

### Phase 4 — only then visible graphing
11. graph assembly service
12. rendering metadata
13. walkthrough read routes
14. evidence overlay rendering

This order matters.
Otherwise the system risks becoming visually impressive before becoming procedurally safe.

## 7. Proposed implementation invariants

Any future code bridge should preserve these invariants:

### Invariant A
No `ResearchNode` becomes executable without a witness-backed transition and explicit guard pass.

### Invariant B
No UI action changes runtime privilege directly.

### Invariant C
No evidence state implies authority automatically.

### Invariant D
No challenge outcome silently overwrites the historical branch.

### Invariant E
Every runtime lock collision must degrade to a safer path, never to speculative continuation.

### Invariant F
Every visible graph node must remain class-separable:
execution / research / witness / cinematic

## 9. What should NOT happen

The bridge will fail if the repo adopts any of the following shortcuts:

- using free-form JSON blobs instead of typed status objects
- allowing UI-driven branch mutation without runtime guards
- storing research and execution artifacts in one undifferentiated pool
- allowing signed records to masquerade as legitimacy
- allowing challenge resolution to overwrite historical witness
- implementing walkthrough first and state algebra later

That path would produce aesthetic intelligence, not accountable continuity.

## 13. Earth paragraph

In a real machine room, the emergency stop, the maintenance backlog, the inspection seal, the operator dashboard, and the fault-state transition table are not the same subsystem.
